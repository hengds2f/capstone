📖 Why Distributed Processing?

When a single machine can't handle the data or compute, we distribute work across a cluster of machines.

FrameworkBest ForSG Use Case
DaskScaling pandas workflowsProcessing 10 years of HDB data
PySparkPetabyte-scale ETLLTA taxi GPS data (billions of rows)

📖 Dask — Parallel Pandas

Python
import dask.dataframe as dd

# Dask reads data lazily — no memory overflow
ddf = dd.read_csv('hdb_resale_*.csv')   # Multiple files as partitions

# Same pandas API — but executed in parallel
result = (
    ddf[ddf['town'] == 'BISHAN']
    .groupby('flat_type')['resale_price']
    .mean()
    .compute()   # ← Triggers actual computation
)
print(result)

# Key Dask concepts:
# 1. Lazy evaluation — nothing runs until .compute()
# 2. Partitions — data is split across workers
# 3. Task graph — Dask builds a DAG of operations

📖 PySpark — Big Data at Scale

Python
from pyspark.sql import SparkSession
from pyspark.sql.functions import avg, count, col

spark = SparkSession.builder.appName("SG_HDB").getOrCreate()

# Read data into a distributed DataFrame
df = spark.read.csv("hdb_resale_full.csv", header=True, inferSchema=True)

# Transformations (lazy — builds execution plan)
result = (
    df.filter(col("town") == "BISHAN")
      .groupBy("flat_type")
      .agg(
          avg("resale_price").alias("avg_price"),
          count("*").alias("count")
      )
      .orderBy("avg_price", ascending=False)
)

# Action (triggers execution across cluster)
result.show()

# +----------+----------+-----+
# | flat_type| avg_price|count|
# +----------+----------+-----+
# |EXECUTIVE | 820000.0 |  45 |
# | 5 ROOM   | 650000.0 | 120 |
# | 4 ROOM   | 480000.0 | 350 |
# | 3 ROOM   | 350000.0 | 200 |

📖 MapReduce Pattern

Python
from functools import reduce
from collections import Counter

# Simulating MapReduce for HDB price analysis
data_partitions = [
    [("BISHAN", 500000), ("TAMPINES", 420000), ("BISHAN", 520000)],
    [("TAMPINES", 430000), ("BISHAN", 510000), ("YISHUN", 350000)],
    [("YISHUN", 360000), ("TAMPINES", 440000), ("YISHUN", 340000)],
]

# MAP phase: Each partition emits (key, value) pairs
def map_fn(partition):
    return [(town, price) for town, price in partition]

# SHUFFLE: Group by key (handled by framework)
def shuffle(mapped_results):
    groups = {}
    for pairs in mapped_results:
        for town, price in pairs:
            groups.setdefault(town, []).append(price)
    return groups

# REDUCE phase: Aggregate values for each key
def reduce_fn(key, values):
    return (key, sum(values) / len(values), len(values))

# Execute MapReduce
mapped = [map_fn(p) for p in data_partitions]          # Parallel
shuffled = shuffle(mapped)                              # Framework
results = [reduce_fn(k, v) for k, v in shuffled.items()] # Parallel

for town, avg_price, count in results:
    print(f"{town}: avg ${avg_price:,.0f} ({count} transactions)")
⚠️ Common Mistakes
  • Using .collect() on large data: Pulls all data to driver — causes OOM. Use .show() or .take(10)
  • Not partitioning properly: Too few partitions = uneven load; too many = overhead
  • Ignoring data skew: If one key (e.g., "WOODLANDS") has 10x more data, that partition becomes a bottleneck
