📖 Star Schema Design

The star schema is the most common data warehouse design pattern. It separates facts (measurable events) from dimensions (descriptive context).

Why star schema? Queries are intuitive — always start from the fact table and JOIN to dimensions. Aggregations are fast because fact tables contain only keys and measures.

📖 Normalization vs Denormalization

Normalization (3NF)Denormalization (Star Schema)
No data redundancyControlled redundancy for speed
More JOINs neededFewer JOINs, simpler queries
Better for OLTP (transactions)Better for OLAP (analytics)
Smaller storageLarger but faster reads

🎯 Our Schema Choices

SQL
-- Normalized: dim_location stores locations once
-- Each transaction references location_id (FK)
-- Benefit: Update town name in one place

-- Denormalized: raw_data stores everything flat
-- No JOINs needed for simple queries
-- Benefit: Faster reads, simpler analysis queries
